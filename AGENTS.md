# Repository Guidelines

## Project Structure & Module Organization
`src/` contains the ROS 2 packages. Core bringup and serial chassis control live in `src/turn_on_traymover_robot`, keyboard teleop in `src/traymover_robot_keyboard`, robot model assets in `src/traymover_robot_description`, navigation in `src/traymover_robot_nav2`, and lidar/SLAM stacks under `src/traymover_lidar_ros2` and `src/traymover_robot_slam`. `reference/` stores hardware notes and upstream examples; `scripts/` holds local utilities. `build/`, `install/`, and `log/` are generated by `colcon` and should not be edited.

## Build, Test, and Development Commands
Run all commands from the workspace root:

- `source /opt/ros/humble/setup.bash` loads the ROS 2 Humble toolchain.
- `colcon build --symlink-install` builds the workspace for local iteration.
- `colcon build --packages-select turn_on_traymover_robot traymover_robot_keyboard` rebuilds only touched packages.
- `source install/setup.bash` overlays the built workspace before `ros2 launch` or `ros2 run`.
- `colcon test --packages-select turn_on_traymover_robot && colcon test-result --verbose` runs the current Python test/lint surface.
- `ros2 launch turn_on_traymover_robot base_serial.launch.py use_imu:=false` plus `ros2 run traymover_robot_keyboard traymover_keyboard` is the fastest end-to-end smoke test.

## Coding Style & Naming Conventions
Use Python 3 with 4-space indentation, `snake_case` for modules/functions, and descriptive ROS parameter names. Keep launch and config filenames lowercase with underscores, for example `traymover_gmapping.launch.py`. C++ packages follow `ament_cmake`; use 2-space indentation and format with the repo `.clang-format` at `src/traymover_robot_slam/ndt_omp_ros2/.clang-format`. Do not hand-edit generated outputs under `build/`, `install/`, or `log/`.

## Testing Guidelines
Place Python tests in package-local `test/` directories and name them `test_*.py`; examples live in `src/turn_on_traymover_robot/test/`. CMake packages should keep `BUILD_TESTING` blocks and `ament_lint_auto` enabled. There is no enforced global coverage threshold in-tree, but every behavior change should add or update regression coverage and leave `colcon test` passing for affected packages.

## Commit & Pull Request Guidelines
Recent history uses short imperative subjects, often with Conventional Commit prefixes such as `fix:` and `feat:`. Keep the subject concise, explain why in the body, and call out hardware-sensitive changes like `/dev/ttyCH*`, lidar IPs, or Nav2/EKF parameter edits. Pull requests should summarize scope, list touched packages, include the test commands run, and attach RViz screenshots or logs for behavior changes that affect bringup, mapping, or navigation.

## Configuration & Hardware Notes
Review `src/turn_on_traymover_robot/config/` and `src/traymover_robot_nav2/param/` before changing defaults. Device names, serial baud rates, and lidar network settings are deployment-specific; verify them locally before committing.
